# Google Maps + Accurate Water Geometries Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the hand-drawn water geometries with real OSM data and migrate the Karte page from Leaflet/OSM tiles to the Google Maps JS API.

**Architecture:** A dev-time Node script (`scripts/fetch-waters.mjs`) queries the Overpass API using a per-water manifest (OSM ids / name queries + section clip points), stitches/clips/simplifies the geometries, and generates `src/data/waters/geometries.ts`. `berlinWaters.ts` keeps only permit-verified metadata and merges in the generated geometries. `karte.tsx` is rewritten on `@vis.gl/react-google-maps`.

**Tech Stack:** React 19, TanStack Router, `@vis.gl/react-google-maps`, Overpass API (dev-time only), vitest.

## Global Constraints

- All user-facing strings localized de/en via `src/i18n/dict.ts` (`LocalizedText`).
- Coordinates in data files are GeoJSON order `[lng, lat]` (see `src/data/waters/types.ts`).
- No runtime fetching of geometry data; the app stays static.
- API key only via `import.meta.env.VITE_GOOGLE_MAPS_API_KEY`; never committed.
- `npm run typecheck`, `npm test`, and `npm run format` must pass at every commit.
- Lake polygons ≤ ~120 points; total generated file should stay well under ~300 KB.

- Test: `scripts/lib/geometry.test.mjs`

**Interfaces:**
- Produces: `stitchWays(ways: {coords: [lng,lat][]}[]): [lng,lat][]` — chains way segments that share endpoints into one line; `stitchRings(ways)` same but returns a closed ring; `clipLine(coords, from: [lng,lat], to: [lng,lat]): [lng,lat][]` — slice of the line between the vertices nearest to `from`/`to` (inclusive, preserving direction); `simplify(coords, maxPoints: number): [lng,lat][]` — Douglas-Peucker, tolerance auto-raised until `length <= maxPoints`.

### Task 2: Overpass fetch script + generated geometries

**Files:**
- Create: `scripts/fetch-waters.mjs` (manifest + fetch + generate)
- Create (generated): `src/data/waters/geometries.ts`

**Interfaces:**
- Consumes: helpers from Task 1.
- Produces: `export const waterGeometries: Record<string, WaterGeometry>` keyed by the existing water ids in `berlinWaters.ts` (all 23).

- [ ] **Step 1: Research OSM objects per water.** For each water, find the OSM way/relation via Overpass (`https://overpass-api.de/api/interpreter`), e.g. `[out:json];way["name"="Landwehrkanal"]["waterway"](52.4,13.2,52.6,13.6);out ids tags;`. For the four unconfirmed lakes (Kiessee, Körner See, Grabensprung, Bucher Teich I–II) cross-check against DAV Berlin water lists / maps before pinning ids. Record findings in the manifest.
- [ ] **Step 2: Write the manifest + script.** Manifest entry shape: `{ id, kind: 'line'|'polygon', select: { name?, ids?, filter? }, clip?: { from: [lng,lat], to: [lng,lat] }, maxPoints? }`. Clip points come straight from the permit sections (Schleuse Plötzensee→Humboldthafen; Lohmühlenbrücke→Britzer Hafen; Seddinsee→Wernsdorfer See; B1→Spree; Einlauf Britzer Zweigkanal→Mühlendammschleuse). Script fetches with 1 req/s throttling, stitches, clips, simplifies, and writes `geometries.ts` with a `// GENERATED by scripts/fetch-waters.mjs — do not edit` header.
- [ ] **Step 3: Run `node scripts/fetch-waters.mjs`** — expect a generated file with 23 entries; `npm run typecheck` passes.
- [ ] **Step 4: Sanity-check output** — each water's coordinates fall inside greater Berlin (lng 12.9–13.9, lat 52.3–52.7); lines have ≥ 10 points; polygons are closed rings. Add these as assertions in the script itself so regeneration self-checks.
